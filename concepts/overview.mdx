## Data model

The Open API Platform organizes everything around two primary resource types.

| Resource | Description |
|---|---|
| **Users** | Represent individual accounts that interact with your application. |
| **Resources** | Generic, application-defined objects that belong to your project. |

Every object in the platform shares a common set of system-managed fields:

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier for the object, assigned by the platform. |
| `created_at` | string (ISO 8601) | Timestamp of when the object was created. |
| `updated_at` | string (ISO 8601) | Timestamp of the most recent update to the object. |

<Note>
  The `id`, `created_at`, and `updated_at` fields are read-only. You cannot set or modify them directly.
</Note>

## RESTful conventions

The Open API Platform follows standard REST conventions so that the API behaves predictably across every endpoint.

- **HTTP methods** — use `GET` to retrieve, `POST` to create, `PUT` to update, and `DELETE` to remove resources.
- **JSON bodies** — send and receive data as JSON. Every request with a body must include the `Content-Type: application/json` header.
- **HTTP status codes** — the API signals success or failure through standard HTTP status codes (for example, `200 OK`, `201 Created`, `404 Not Found`). See [Requests & Responses](/concepts/request-response) for the full list.
- **Base URL** — all endpoints are relative to `https://api.example.com/v1`.
