MC OpenAPI is a JSON-over-HTTPS API. Every call targets a gateway host under `/openapi/v1/`, carries the signature v2 headers, and â€” where applicable â€” a JSON body. Every response is a JSON envelope with `code`, `msg`, `data`, and `success`.

## Request format

### Required headers

Include the following headers on every request:

| Header | Required | Value |
| --- | --- | --- |
| `X-Access-Key` | Yes | Your Access Key ID |
| `X-Signature` | Yes | HMAC-SHA256 signature over the canonical string (**Base64**) |
| `X-Timestamp` | Yes | Current Unix time in **milliseconds** |
| `X-Nonce` | Yes | Unique per request (UUID v4 or â‰Ą32 hex chars) |
| `Content-Type` | Recommended for `POST` / `PUT` / `PATCH` | `application/json` |
| `Accept` | Recommended | `application/json` |
| `X-Signature-Version` | Optional | Suggested value `2` (troubleshooting) |

<Warning>
  Requests without valid signing headers or `X-Nonce`, or with a `X-Timestamp` outside the allowed window, are rejected. Never commit your Secret Key to source control or expose it in client-side code.
</Warning>

See [Authentication](/authentication) for the exact signature **v2** format and reference implementations in Python, Java, and Node.js.

### String to sign (v2 summary)

```text
{HTTP_METHOD}
{REQUEST_URI}
{QUERY_STRING}
{SIGNED_HEADERS}
{TIMESTAMP}
{NONCE}
[{BODY_SHA256_HEX}]
```

- Omit the entire `QUERY_STRING` line when there is no query (do **not** insert a blank line).
- `SIGNED_HEADERS` includes `x-access-key`, `x-nonce`, and `x-timestamp` (lowercase keys, sorted).
- For `POST`/`PUT`/`PATCH` with a non-empty body, append the SHA-256 of the raw body bytes as **lowercase hex**.
- `X-Signature` is Base64, not hex.

### Example: query the USD exchange rates

```bash
curl --request GET \
  --url "https://gateway.mcconnects.com/openapi/v1/mc-account/exchange-rate/query/usd" \
  --header "X-Access-Key: $MC_ACCESS_KEY" \
  --header "X-Signature: $MC_SIGNATURE" \
  --header "X-Timestamp: $MC_TIMESTAMP" \
  --header "X-Nonce: $MC_NONCE" \
  --header "Accept: application/json"
```

## Response format

All responses are JSON. The MC OpenAPI wraps every result in a common envelope.

### Response envelope

```json
{
  "code": "0",
  "msg": "success",
  "data": { },
  "success": true
}
```

<ResponseField name="code" type="string">
  Business status code. `"0"` indicates success. Any other value indicates a business-level error â€” see [Error Handling](/concepts/errors).
</ResponseField>

<ResponseField name="msg" type="string">
  Human-readable status message. For successful calls this is typically `"success"`. On failure it describes the reason.
</ResponseField>

<ResponseField name="data" type="object | array | null">
  The actual response payload. Shape depends on the endpoint â€” a single object, a list, or a paginated page object.
</ResponseField>

<ResponseField name="success" type="boolean">
  Convenience flag; `true` when `code == "0"`, otherwise `false`.
</ResponseField>

### Paginated list response

Endpoints that return a page of results place a page object inside `data`:

```json
{
  "code": "0",
  "msg": "success",
  "success": true,
  "data": {
    "pageNum": 1,
    "pageSize": 20,
    "total": 132,
    "pages": 7,
    "list": [
      { "transactionId": "T-1001", "amount": "125.50", "currency": "USDT" }
    ]
  }
}
```

See [Pagination](/concepts/pagination) for how to request additional pages.

## HTTP status codes

MC OpenAPI uses standard HTTP status codes for transport-level outcomes. Even on a `200 OK`, always inspect the envelope `code` for the business result.

| Code | Meaning |
| --- | --- |
| `200` | Request delivered successfully. Check `code` in the JSON envelope for the business result. |
| `400` | Malformed request â€” missing parameters or invalid JSON body. |
| `401` | Signature invalid, `X-Timestamp` skewed, or Access Key not recognised. |
| `403` | Access Key is valid but lacks the required scope (`READ`, `TRADE`, or `WITHDRAW`). |
| `404` | The endpoint or resource does not exist. |
| `429` | Rate limit exceeded â€” see [Rate Limits](/guides/rate-limits). |
| `500` | Unexpected gateway error. Retry with backoff. |
| `503` | Gateway temporarily unavailable. Retry with backoff. |

<Tip>
  Log the `code`, `msg`, and any correlation ID from response headers on every non-`200` response â€” it makes support requests much faster to triage.
</Tip>
