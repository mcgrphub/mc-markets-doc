MC OpenAPI reports failures through **two layers**: a standard HTTP status code, and business fields inside the JSON envelope (`code` / `codeKey`). Robust integrations check both.

## Two-layer error model

1. **Transport layer** ÔÇö the HTTP status code. `4xx` indicates a client problem, `5xx` a gateway problem.
2. **Business layer** ÔÇö `code` and `codeKey` in the JSON envelope. A `200 OK` HTTP response can still carry a business error such as insufficient balance or a symbol not found.

Always inspect `success` / `code` / `codeKey` even when the HTTP status is `200`.

## Error response format

Failed calls keep the same envelope shape, with `success: false` and a non-success business `code`:

```json
{
  "code": 1000005,
  "codeKey": "PARAM_ERROR",
  "msg": "Invalid request parameter",
  "success": false,
  "data": null
}
```

<ResponseField name="code" type="integer | string">
  Business status code. Success responses typically use `0` / `200` depending on the envelope path. Failures use a **7-digit** error code.
</ResponseField>

<ResponseField name="codeKey" type="string">
  Stable error key, for example `PARAM_ERROR` or `SIGN_ERROR`. Prefer branching on `codeKey` or `code`, not on `msg`.
</ResponseField>

<ResponseField name="msg" type="string">
  Human-readable message. Suitable for logs; do not branch on the exact string.
</ResponseField>

<ResponseField name="success" type="boolean">
  `false` on business failures.
</ResponseField>

<ResponseField name="data" type="object | null">
  Usually `null` on errors. Some validation errors include a `data` object with per-field detail.
</ResponseField>

## HTTP status codes

| Code | When it occurs | What to do |
| --- | --- | --- |
| `400` | Malformed JSON, missing required parameter | Fix the request and retry. |
| `401` | Missing / invalid `X-Signature`, or `X-Timestamp` skewed | Re-check the signing algorithm and clock; re-sign and retry. |
| `403` | Access Key lacks the required scope | Add `READ` / `TRADE` / `WITHDRAW` in the key management console. |
| `404` | Path is wrong, or the resource ID does not exist | Verify the URL and IDs. |
| `429` | Rate limit exceeded | Back off ÔÇö see [Rate Limits](/guides/rate-limits). |
| `500` / `502` / `503` | Gateway or upstream error | Retry with exponential backoff. |

## Common business codes

Full catalog: [Error codes](/api-reference/error-codes) (public OpenAPI-reachable codes only).

| `codeKey` | `code` | Meaning | Suggested action |
| --- | --- | --- | --- |
| `PARAM_ERROR` | `1000005` | Invalid request parameter | Fix parameters per `msg`. |
| `NOT_EXIST` | `1000002` | Resource does not exist | Verify IDs in the request. |
| `ACCOUNT_BALANCE_INSUFFICIENT` | `2004003` | Insufficient balance | Top up or reduce the amount. |
| `ORDER_OPERATION_FAILED` | `2100001` | Order operation failed | Inspect `msg`; retry only if transient. |
| `DEFAULT` | `1000000` | Internal server error | Retry with backoff; contact support if it persists. |

## Handling errors

### Python

```python
import requests, time

def call(method, path, body=""):
    # ... build signed request as in the auth docs ...
    response = requests.request(method, url, headers=headers, data=body)

    if response.status_code == 429 or 500 <= response.status_code < 600:
        raise TransientError(response.status_code, response.text)

    payload = response.json()
    if payload.get("success") is True:
        return payload.get("data")

    code_key = payload.get("codeKey") or str(payload.get("code"))
    if response.status_code == 401:
        raise AuthError(code_key, payload.get("msg") or payload.get("message"))

    raise BusinessError(code_key, payload.get("msg"))


def with_backoff(fn, *args, max_retries=5, **kwargs):
    delay = 1.0
    for attempt in range(max_retries):
        try:
            return fn(*args, **kwargs)
        except TransientError:
            if attempt == max_retries - 1:
                raise
            time.sleep(delay)
            delay = min(delay * 2, 30)
```

### JavaScript

```javascript
async function call(method, path, body = "") {
  // ... build signed request as in the auth docs ...
  const res = await fetch(url, { method, headers, body: body || undefined });

  if (res.status === 429 || res.status >= 500) {
    throw new TransientError(res.status, await res.text());
  }

  const payload = await res.json();
  if (payload.success === true) return payload.data;

  const codeKey = payload.codeKey || String(payload.code);
  if (res.status === 401) {
    throw new AuthError(codeKey, payload.msg || payload.message);
  }
  throw new BusinessError(codeKey, payload.msg);
}
```

<Tip>
  For `429` and `5xx` responses, use **exponential backoff with jitter**: start at ~1 s, double on each failure, cap at ~30 s, and add a small random offset to avoid synchronized retries.
</Tip>
