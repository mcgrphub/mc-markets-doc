Whenever a request cannot be completed successfully, the Open API Platform returns a structured JSON error response with a consistent shape. Instead of relying solely on HTTP status codes, you can inspect the response body for a machine-readable error code and a human-readable message that pinpoint exactly what went wrong.

## Error response format

Every error response follows the same envelope structure:

```json
{
  "error": {
    "code": "validation_error",
    "message": "The 'email' field is required.",
    "details": [
      {
        "field": "email",
        "message": "This field is required."
      }
    ]
  }
}
```

The fields within the `error` object are:

<ResponseField name="error.code" type="string">
  A stable, machine-readable identifier for the error type. Use this field — not the `message` — to drive error-handling logic in your code, because messages may change without notice.
</ResponseField>

<ResponseField name="error.message" type="string">
  A human-readable description of the error, suitable for logging or displaying to developers. Do not rely on this string for programmatic branching.
</ResponseField>

<ResponseField name="error.details" type="array | null">
  An optional array of objects that provide field-level context for validation errors. Each item contains a `field` (the name of the invalid field) and a `message` (a description of why validation failed). Returns `null` or is omitted for non-validation errors.
</ResponseField>

## Error codes

The table below lists every error code the API returns, the HTTP status it accompanies, and what it means for your integration.

| Code | HTTP Status | Description |
|---|---|---|
| `authentication_required` | `401` | No API key was provided in the request. Add an `Authorization: Bearer YOUR_API_KEY` header. |
| `invalid_api_key` | `401` | The API key you provided is invalid, revoked, or expired. Generate a new key and retry. |
| `insufficient_permissions` | `403` | Your API key does not have the scope required for this action. Check the key's permissions in your dashboard. |
| `not_found` | `404` | The resource you requested does not exist, or you do not have access to it. Verify the ID in your request. |
| `validation_error` | `422` | The request body passed format checks but failed business-logic validation. Inspect `error.details` for field-level information. |
| `rate_limit_exceeded` | `429` | You have exceeded the allowed request rate. Wait before retrying, and consider implementing exponential backoff. |
| `internal_error` | `500` | An unexpected error occurred on the platform. Retry the request after a short delay. If the problem persists, contact support. |

<Note>
  The `not_found` error is intentionally returned for both "does not exist" and "you lack access" cases to avoid leaking the existence of private resources.
</Note>

## Handling errors

The examples below show a robust pattern for catching and responding to API errors in your application.

### Python

```python
import requests
from time import sleep

API_KEY = "YOUR_API_KEY"
BASE_URL = "https://api.example.com/v1"

def create_user(name: str, email: str) -> dict:
    headers = {
        "Authorization": f"Bearer {API_KEY}",
        "Content-Type": "application/json",
        "Accept": "application/json",
    }
    payload = {"name": name, "email": email}

    try:
        response = requests.post(
            f"{BASE_URL}/users",
            headers=headers,
            json=payload,
        )
        response.raise_for_status()
        return response.json()["data"]

    except requests.HTTPError as exc:
        error_body = exc.response.json().get("error", {})
        code = error_body.get("code", "unknown")
        message = error_body.get("message", "An unexpected error occurred.")

        if code == "validation_error":
            details = error_body.get("details", [])
            for detail in details:
                print(f"Validation failed on '{detail['field']}': {detail['message']}")
        elif code == "rate_limit_exceeded":
            print("Rate limit hit. Waiting 60 seconds before retrying...")
            sleep(60)
            return create_user(name, email)  # Retry once
        elif code == "invalid_api_key":
            print("Invalid API key. Check your credentials.")
        else:
            print(f"API error [{code}]: {message}")

        raise
```

### JavaScript

```javascript
const BASE_URL = "https://api.example.com/v1";
const API_KEY = "YOUR_API_KEY";

async function createUser(name, email) {
  const headers = {
    Authorization: `Bearer ${API_KEY}`,
    "Content-Type": "application/json",
    Accept: "application/json",
  };

  try {
    const response = await fetch(`${BASE_URL}/users`, {
      method: "POST",
      headers,
      body: JSON.stringify({ name, email }),
    });

    if (!response.ok) {
      const body = await response.json();
      const { code, message, details } = body.error ?? {};

      switch (code) {
        case "validation_error":
          details?.forEach(({ field, message: msg }) => {
            console.error(`Validation failed on '${field}': ${msg}`);
          });
          break;
        case "rate_limit_exceeded":
          console.warn("Rate limit hit. Implement backoff before retrying.");
          break;
        case "invalid_api_key":
          console.error("Invalid API key. Check your credentials.");
          break;
        default:
          console.error(`API error [${code}]: ${message}`);
      }

      throw new Error(`Request failed: ${code ?? response.status}`);
    }

    const body = await response.json();
    return body.data;
  } catch (err) {
    // Re-throw so the caller can handle or log the error
    throw err;
  }
}
```

<Tip>
  For `429 Rate Limit Exceeded` and `5xx` server errors, implement **exponential backoff with jitter**: wait an initial delay (for example, 1 second), double it on each successive failure, and add a small random jitter to avoid thundering-herd problems. Most transient server errors resolve within a few retries.
</Tip>
