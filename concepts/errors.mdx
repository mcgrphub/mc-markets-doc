MC OpenAPI reports failures through **two layers**: a standard HTTP status code, and a business `code` field inside the JSON envelope. Robust integrations check both.

## Two-layer error model

1. **Transport layer** — the HTTP status code. `4xx` indicates a client problem, `5xx` a gateway problem.
2. **Business layer** — the `code` field in the JSON envelope. A `200 OK` HTTP response can still carry a business error such as insufficient balance or a symbol not found.

Always inspect `code` even when the HTTP status is `200`.

## Error response format

Failed calls return the same envelope shape as successful ones, with a non-`"0"` `code` and `success: false`:

```json
{
  "code": "40001",
  "msg": "Invalid signature",
  "success": false,
  "data": null
}
```

<ResponseField name="code" type="string">
  Business status code. `"0"` is success. Non-zero values identify the specific error class — use this to drive retry / user-facing logic.
</ResponseField>

<ResponseField name="msg" type="string">
  Human-readable message. Suitable for logs; do not branch on the exact string.
</ResponseField>

<ResponseField name="success" type="boolean">
  `false` on any non-`"0"` `code`.
</ResponseField>

<ResponseField name="data" type="object | null">
  Usually `null` on errors. Some validation errors include a `data` object with per-field detail.
</ResponseField>

## HTTP status codes

| Code | When it occurs | What to do |
| --- | --- | --- |
| `400` | Malformed JSON, missing required parameter | Fix the request and retry. |
| `401` | Missing / invalid `X-Signature`, or `X-Timestamp` skewed | Re-check the signing algorithm and clock; re-sign and retry. |
| `403` | Access Key lacks the required scope | Add `READ` / `TRADE` / `WITHDRAW` in the key management console. |
| `404` | Path is wrong, or the resource ID does not exist | Verify the URL and IDs. |
| `429` | Rate limit exceeded | Back off — see [Rate Limits](/guides/rate-limits). |
| `500` / `502` / `503` | Gateway or upstream error | Retry with exponential backoff. |

## Common business codes

The table below lists frequently-seen business `code` values. This is not exhaustive — new codes are added over time; treat unknown codes as retryable / user-facing errors based on context.

| `code` | Meaning | Suggested action |
| --- | --- | --- |
| `"0"` | Success | Consume `data`. |
| `40001` | Invalid signature | Recompute signature; verify Secret Key and canonical string. |
| `40002` | Timestamp expired or skewed | Sync clock; use current UTC ms. |
| `40003` | Access Key not found or disabled | Check key status in the console. |
| `40301` | Insufficient permission | Add the required scope to the key. |
| `40401` | Resource not found | Verify IDs in the request. |
| `42201` | Parameter validation failed | Fix parameters per `msg`. |
| `42901` | Rate limit exceeded | Back off and retry. |
| `50001` | Internal error | Retry with backoff; contact support if it persists. |

<Note>
  Codes are strings, not integers — always compare with string equality (`code === "0"`).
</Note>

## Handling errors

### Python

```python
import requests, time

def call(method, path, body=""):
    # ... build signed request as in the auth docs ...
    response = requests.request(method, url, headers=headers, data=body)

    if response.status_code == 429 or 500 <= response.status_code < 600:
        raise TransientError(response.status_code, response.text)

    payload = response.json()
    code = payload.get("code")

    if code == "0":
        return payload["data"]

    if code in {"40001", "40002"}:
        raise AuthError(code, payload["msg"])
    if code == "42901":
        raise TransientError(code, payload["msg"])

    raise BusinessError(code, payload["msg"])


def with_backoff(fn, *args, max_retries=5, **kwargs):
    delay = 1.0
    for attempt in range(max_retries):
        try:
            return fn(*args, **kwargs)
        except TransientError:
            if attempt == max_retries - 1:
                raise
            time.sleep(delay)
            delay = min(delay * 2, 30)
```

### JavaScript

```javascript
async function call(method, path, body = "") {
  // ... build signed request as in the auth docs ...
  const res = await fetch(url, { method, headers, body: body || undefined });

  if (res.status === 429 || res.status >= 500) {
    throw new TransientError(res.status, await res.text());
  }

  const payload = await res.json();

  if (payload.code === "0") return payload.data;

  if (payload.code === "40001" || payload.code === "40002") {
    throw new AuthError(payload.code, payload.msg);
  }
  if (payload.code === "42901") {
    throw new TransientError(payload.code, payload.msg);
  }
  throw new BusinessError(payload.code, payload.msg);
}
```

<Tip>
  For `429` and `5xx` responses, use **exponential backoff with jitter**: start at ~1 s, double on each failure, cap at ~30 s, and add a small random offset to avoid synchronized retries.
</Tip>
