---

If something isn't working as expected, this guide walks you through step-by-step diagnosis and fixes for the most common integration problems. Work through the relevant section below to get back on track quickly.

<AccordionGroup>
  <Accordion title="401 Unauthorized">
    A `401` response means the server could not authenticate your request. Work through this checklist:

    - **Check your Authorization header format.** The API requires a Bearer token — not Basic auth. Your header must look exactly like this:

      ```http
      Authorization: Bearer <your_api_key>
      ```

    - **Check for extra whitespace.** Copy-paste errors often introduce a leading or trailing space in the key. Trim the value before sending.

    - **Check whether the key has been revoked.** Go to **Settings → API Keys** in your dashboard. If the key shows a **Revoked** status, it can no longer authenticate requests. Click **Create New Key** to generate a replacement, then update your environment variables or secrets manager with the new value.
  </Accordion>

  <Accordion title="403 Forbidden">
    A `403` response means your API key is valid and authenticated, but it **lacks the required scope** for the endpoint you're calling.

    To fix this:

    1. Go to **Settings → API Keys** in your dashboard.
    2. Find the key you're using and open its details.
    3. Review the **Assigned Scopes** list. If the required scope is missing, edit the key to add it — or create a new key with the correct scopes for that integration.

    Each endpoint's reference page lists the minimum required scope under the **Authentication** section.
  </Accordion>

  <Accordion title="404 Not Found">
    A `404` response means the server couldn't locate the resource you requested. Check the following:

    - **Verify the resource ID.** Confirm that the ID in your request path (e.g. `res_abc123`) actually exists and belongs to your account. IDs are case-sensitive.
    - **Check the API version prefix.** All endpoints must include `/v1/` in the path:

      ```
      GET https://api.example.com/v1/resources/{id}
      ```

      Omitting the version segment or using an unsupported version (e.g. `/v0/`) will return a `404`.
    - **Check for typos in the endpoint path.** Compare your URL against the [API reference](/api-reference) to make sure every segment is spelled correctly.
  </Accordion>

  <Accordion title="422 Validation Error">
    A `422 Unprocessable Entity` response means your request was received and understood, but one or more fields failed validation. The response body always includes an `error.details` array that identifies exactly what went wrong:

    ```json
    {
      "error": {
        "code": "validation_error",
        "message": "Request validation failed.",
        "details": [
          {
            "field": "email",
            "issue": "Must be a valid email address."
          },
          {
            "field": "amount",
            "issue": "Required field is missing."
          }
        ]
      }
    }
    ```

    For each item in `error.details`:

    - Confirm the field is **present** in your request body if it's required.
    - Confirm the **data type** matches what the API expects (e.g. integer vs. string, ISO 8601 date format).
    - Check for **null values** on non-nullable fields.
  </Accordion>

  <Accordion title="429 Rate Limit Exceeded">
    A `429 Too Many Requests` response means you've hit the rate limit for your API key within the current rolling window.

    **Steps to handle this gracefully:**

    1. Read the `Retry-After` response header. It contains the number of seconds you must wait before making another request:

       ```http
       Retry-After: 12
       ```

    2. Implement **exponential backoff** in your retry logic. Instead of retrying immediately, wait progressively longer between each attempt (e.g. 1 s, 2 s, 4 s, 8 s) and add a small random jitter to avoid synchronized retry storms.

    3. Monitor the `X-RateLimit-Remaining` header on successful responses so you can slow down proactively before hitting the limit.

    For limit values by tier and a backoff implementation example, see the [Rate Limits guide](/guides/rate-limits).
  </Accordion>

  <Accordion title="500 Internal Server Error">
    A `500` response indicates an unexpected error on the server side. This is almost always transient.

    **What to do:**

    1. Wait a few seconds and retry the request using **exponential backoff**.
    2. Check the [status page](https://status.example.com) to see whether there's a known incident affecting the API.
    3. If the error persists after several retries, contact support. Include the **`X-Request-ID`** value from the response headers in your message — this lets the team trace your exact request in the server logs:

       ```http
       X-Request-ID: req_7f3a2b1c9e
       ```
  </Accordion>

  <Accordion title="Webhook not being received">
    If your registered endpoint isn't receiving webhook events, work through the following checks:

    - **Your endpoint must return a `2xx` status code.** If it returns anything else (including a redirect), the platform marks the delivery as failed and retries according to the retry schedule.
    - **Check your firewall and network rules.** Inbound HTTPS traffic on port 443 must be allowed from the Open API Platform's webhook IP ranges (listed in the dashboard under **Settings → Webhooks**).
    - **Verify the registered URL is correct.** Go to **Settings → Webhooks** and confirm the URL exactly matches your publicly accessible endpoint — including the correct protocol (`https://`), domain, and path.
    - **Check your server logs** for incoming `POST` requests to rule out delivery attempts that your application is silently rejecting.
  </Accordion>

  <Accordion title="CORS errors in browser">
    If you're making API requests directly from browser-side JavaScript, you will encounter **CORS (Cross-Origin Resource Sharing) errors** — and more critically, your API key will be **exposed to anyone who inspects your page's network traffic**.

    **The correct approach is to proxy all API requests through your own backend:**

    1. Your frontend calls your own server endpoint (e.g. `POST /api/my-action`).
    2. Your server makes the authenticated request to the Open API Platform using the API key stored securely in an environment variable.
    3. Your server returns only the data the frontend needs.

    This pattern keeps your API key private and eliminates CORS issues entirely, since same-origin requests from your frontend to your own backend are unrestricted.
  </Accordion>
</AccordionGroup>

<Info>
  When contacting support about a specific request, always include the **`X-Request-ID`** header value from the response. This unique identifier lets the support team locate your exact request in the server logs and dramatically speeds up diagnosis.
</Info>
