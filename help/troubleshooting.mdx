Use this guide as a decision tree when a request fails. Always inspect **both** the HTTP status and the `code` field in the JSON envelope before diagnosing.

## HTTP 401 — signature or timestamp problem

```json
{ "code": "40001", "msg": "Invalid signature", "success": false }
```

### Checklist

- **Canonical string.** Print exactly what you signed. It must be five lines: `METHOD\nPATH\nSORTED_QUERY_STRING\nBODY_OR_EMPTY\nTIMESTAMP`. Empty query string or body means an empty line, not the string `"null"`.
- **Body match.** Serialize the JSON body **once** and reuse the same string for both signing and the HTTP request. `JSON.stringify(obj)` after signing gives a subtly different string than the one you signed.
- **Query ordering.** Sort query parameters by key ascending, joined `key=value` with `&`. Do not URL-encode differently between signing and sending.
- **Method / path case.** Method is uppercase (`GET`, not `get`). Path starts with `/openapi/v1/` and does not include the base URL or the query string.
- **Correct key.** `X-Access-Key` is the Access Key; sign with the Secret Key. It is easy to swap them.
- **Digest format.** `X-Signature` is the **lowercase hex** encoding of the HMAC-SHA256 digest, not base64.

```json
{ "code": "40002", "msg": "Timestamp expired", "success": false }
```

- `X-Timestamp` must be current UTC time in **milliseconds** (13 digits, e.g. `1731657600000`), not seconds.
- Sync your machine clock via NTP. A drift of only a few seconds can trigger this error.
- If you generate `ts` once and reuse it for a batch of requests, re-generate for every call.

## HTTP 403 — insufficient permission

```json
{ "code": "40301", "msg": "Permission denied", "success": false }
```

- Your Access Key exists and the signature is valid, but the key does not have the required scope.
- Open **API Management** in the MC console and add `READ`, `TRADE`, or `WITHDRAW` as required by the endpoint.
- Required scopes are documented on each endpoint page under [API Reference](/api-reference/overview).

## HTTP 404 — endpoint or resource not found

- Verify the full path against [API Reference](/api-reference/overview). All paths start with `/openapi/v1/...` and are case-sensitive.
- Check any path parameters (`{id}`, `{symbolType}`, `{baseCurrency}`, ...) — they must be URL-safe and correspond to real resources.
- If the resource used to exist but no longer does (e.g. a soft-deleted symbol), the endpoint returns `404` even for authenticated requests.

## HTTP 400 — malformed request

```json
{ "code": "42201", "msg": "Parameter validation failed", "success": false }
```

- The `Content-Type: application/json` header is missing on a `POST`/`PUT` with a JSON body.
- Required fields are missing from the body, or types don't match (e.g. sending a number where the schema expects a string).
- Enum values (permission, symbol type, currency) don't match the allowed set — check the endpoint page for the enum values.

## HTTP 429 — rate limited

```json
{ "code": "42901", "msg": "Too many requests", "success": false }
```

- Back off with exponential delay \+ jitter — see [Rate Limits](/guides/rate-limits) for a copy-pasteable retry helper.
- If you routinely hit `429` under normal load, reduce polling frequency or contact support about a higher tier.

## HTTP 5xx — gateway error

- `500` / `502` / `503` are transient. Retry with the same backoff strategy as `429`.
- If a single endpoint returns `5xx` persistently, capture the request timestamp and any correlation header, then contact support.

## Business error with HTTP 200

Do not assume `200 OK` means success. Always check `code`:

```json
{ "code": "60001", "msg": "Insufficient balance", "success": false, "data": null }
```

Typical categories:

- Balance / limit violations on finance-account deposit / withdrawal.
- Symbol suspended or delisted on trading endpoints.
- PoR snapshot not yet generated for the current window on `/openapi/v1/fund/account/por`.

## Debugging checklist

When you file a support request, include the following. It roughly quarters the round-trip time to a resolution:

1. **Environment** — test or production.
2. **Access Key ID** — the public identifier only, never the Secret Key.
3. **Timestamp** — the `X-Timestamp` value on the failing request.
4. **Full path with query string.**
5. **Response body** (Secret Key redacted; other data intact).
6. **Canonical string** you signed (last line = the same `X-Timestamp`).

<Warning>
  Never share your Secret Key when asking for help — not with support, not with colleagues, not in issue trackers. If a Secret Key has been exposed, disable the key immediately and generate a new one.
</Warning>
