MC OpenAPI 采用 URL 路径版本管理。将版本内嵌在路径中可确保你知道集成的是哪个契约,且 API 更新不会静默改变现有请求行为。

## 指定版本

版本位于 Base URL 之后、资源之前,格式为 `/openapi/{version}/{resource}`。

例如:

- `/openapi/v1/mc-account/finance/account/overview`
- `/openapi/v2/mc-account/position/spot/list`(v2 资源)

本参考中未特别标注时均为 `v1`。

## 当前版本

| 版本 | 状态 | 说明 |
| --- | --- | --- |
| v1 | 当前 | 稳定,推荐使用 |
| v2 | 部分可用 | 仅个别资源提供(如 现货持仓、资金费历史) |

<Note>
  v2 目前仅在部分资源上使用,与 v1 不存在完整对应关系。
</Note>

## 废弃策略

<Info>
  废弃版本在移除前至少提前 **12 个月** 公告。破坏性变更不会引入稳定版本——如需破坏兼容性,会以新版本引入。
</Info>

当一个版本进入废弃阶段:

1. 向商户邮箱发送通知。
2. 该版本下的每个响应将包含 `Deprecation` 响应头,含计划移除日期。
3. 在剩余 6、3、1 个月时发送提醒。

## 什么是破坏性变更

以下变动属于破坏性,不会引入稳定版本:

- 删除现有接口或响应字段
- 修改现有字段类型(如 `string` → `integer`)
- 向现有接口新增**必填**参数
- 修改已有场景的 HTTP 状态码

## 非破坏性变更

以下为兼容变动,集成可无需更新:

- 向响应中新增**可选**字段
- 向现有接口新增**可选**参数
- 新增全新接口

建议集成时采用防御性解析——忽略未知字段,而不是失败,以便 API 演进时你的代码仍能稳定工作。
