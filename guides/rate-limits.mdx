MC OpenAPI enforces per-Access-Key request-rate limits at the gateway to protect the platform and ensure fair capacity for every integration.

<Note>
  Exact per-endpoint limits depend on your account tier. If you need higher throughput for a production integration, contact MC support.
</Note>

## How limits are applied

- Limits are counted **per Access Key**, not per source IP.
- Both the test and production gateways enforce their own independent limits.
- Some heavy endpoints (paginated history queries, PoR snapshots) have stricter limits than lightweight metadata endpoints.

## Detecting throttling

When you exceed a limit, the gateway responds with **HTTP `429 Too Many Requests`** and a business error envelope:

```json
{
  "code": "42901",
  "msg": "Too many requests",
  "success": false,
  "data": null
}
```

Always check both signals:

- HTTP status code — reliable transport-level indicator.
- The `code` field in the JSON envelope — `"42901"` (or another rate-limit code documented alongside the endpoint) indicates throttling.

## Recommended handling: exponential backoff with jitter

When you get a `429`, do **not** retry immediately in a tight loop. Wait, then retry with an exponentially growing delay plus a small random jitter to avoid synchronized retry storms.

### Python

```python
import time, random, requests

def call_with_backoff(send_request, *, max_retries=6, base=1.0, cap=30.0):
    for attempt in range(max_retries):
        response = send_request()
        if response.status_code != 429:
            return response

        payload = {}
        try:
            payload = response.json()
        except ValueError:
            pass
        if payload.get("code") not in {"42901", None}:
            return response  # Not a rate-limit error

        delay = min(cap, base * (2 ** attempt)) + random.uniform(0, 0.5)
        time.sleep(delay)

    raise RuntimeError("Exceeded max retries after repeated 429 responses")
```

### JavaScript

```javascript
async function callWithBackoff(sendRequest, { maxRetries = 6, base = 1000, cap = 30_000 } = {}) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const res = await sendRequest();
    if (res.status !== 429) return res;

    const delay = Math.min(cap, base * 2 ** attempt) + Math.floor(Math.random() * 500);
    await new Promise((r) => setTimeout(r, delay));
  }
  throw new Error("Exceeded max retries after repeated 429 responses");
}
```

## Best practices

- **Batch and page efficiently.** Use the largest `pageSize` an endpoint allows — see [Pagination](/concepts/pagination).
- **Cache slow-changing data.** Symbol lists, exchange metadata, and USD rate tables change slowly; cache them locally for at least a few minutes.
- **Separate keys per workload.** Use one Access Key for interactive queries and another for background jobs so a bursty job doesn't starve user traffic.
- **Sync your clocks.** A skewed clock produces `401` signing failures, not `429` — but you'll waste your rate budget on retries. Use NTP.
- **Handle `5xx` the same way.** For `500`, `502`, and `503`, use the same exponential-backoff strategy.

<Tip>
  Log the response `code`, `msg`, and any correlation IDs on every `429`. If throttling is frequent even under normal load, contact support with those log entries to review your tier.
</Tip>
