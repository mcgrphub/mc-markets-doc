<Note>
  MC OpenAPI does not currently deliver webhooks or push events. Integrations that need to detect new activity should poll the relevant list endpoints on a schedule using the patterns below.
</Note>

## What to poll

Choose the endpoint that most tightly matches the event you care about:

| You want to detect | Poll this endpoint | Suggested frequency |
| --- | --- | --- |
| New transaction records | `POST /openapi/v1/transaction/transaction/record/page` | 30–60 s |
| Balance / deposit / withdrawal changes | `POST /openapi/v1/finance/account/overview` | 60 s |
| Spot position changes | `GET /openapi/v1/fund/account/spot/positions/v2` | 60–300 s |
| Funding-fee events | `POST /openapi/v1/funding-fee-record/page/v2` | 300 s (settled every 8 h upstream) |
| Exchange-rate updates | `GET /openapi/v1/exchange-rate/query/usd` | 60–300 s |

See the [API Reference](/api-reference/overview) for the full request/response shape of each endpoint.

## Efficient polling pattern

The goal is to fetch **only what is new** since the last poll and never re-download the entire history.

1. Record a **watermark** — the highest `createdTime` (or `transactionId`) you have already processed.
2. On each poll, request records with `startTime = watermark + 1` (or an equivalent filter).
3. Iterate pages if `pages > 1` — see [Pagination](/concepts/pagination).
4. Persist the new watermark **only after** you have durably processed the batch (write to your DB, publish to your queue, etc.). This avoids missing records if your worker crashes mid-batch.
5. Sleep, then repeat.

### Python example

```python
import time, hmac, hashlib, json, requests

BASE_URL = "https://gateway.mcconnects.com"
ACCESS_KEY, SECRET_KEY = "…", "…"

def sign(method, path, query, body, ts):
    canonical = f"{method}\n{path}\n{query}\n{body}\n{ts}"
    return hmac.new(SECRET_KEY.encode(), canonical.encode(), hashlib.sha256).hexdigest()

def poll_new_transactions(watermark_ms: int, page_size: int = 100):
    path = "/openapi/v1/transaction/transaction/record/page"
    page_num = 1
    new_watermark = watermark_ms
    collected = []

    while True:
        body = json.dumps({
            "pageNum": page_num, "pageSize": page_size,
            "startTime": watermark_ms + 1,
        }, separators=(",", ":"))
        ts = str(int(time.time() * 1000))
        signature = sign("POST", path, "", body, ts)
        r = requests.post(
            BASE_URL + path,
            headers={
                "X-Access-Key": ACCESS_KEY,
                "X-Signature": signature,
                "X-Timestamp": ts,
                "Content-Type": "application/json",
            },
            data=body,
        )
        r.raise_for_status()
        page = r.json()["data"]

        for record in page["list"]:
            collected.append(record)
            new_watermark = max(new_watermark, record["createdTime"])

        if page_num >= page["pages"]:
            break
        page_num += 1

    return collected, new_watermark


watermark = load_watermark_from_db()
while True:
    records, watermark = poll_new_transactions(watermark)
    if records:
        handle_new_records(records)   # your business logic
        persist_watermark(watermark)  # only after successful processing
    time.sleep(30)
```

## Best practices

- **Never poll faster than you need.** Every extra call eats into your [rate limit](/guides/rate-limits) budget.
- **Add jitter** to your polling interval (`sleep(30 + random(0, 5))`) so multiple workers don't hit the gateway in lockstep.
- **Persist watermarks durably.** In-memory state is lost on restart and can cause duplicate or missed records.
- **Deduplicate on the receiving side.** Even with watermarks, network retries and eventual consistency can produce the same record twice — use `transactionId` (or the endpoint's natural key) as an idempotency key.
- **Back off on `429` / `5xx`.** Reuse the [exponential-backoff helper](/guides/rate-limits#recommended-handling-exponential-backoff-with-jitter) for every poll.
